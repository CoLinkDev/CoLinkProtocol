# Remote Camera Stream

Stream live video from a host device's camera to a controller device.

## Design Overview

- **Control plane**: lightweight JSON signaling on the main WebSocket (`list / list-result / open / open-ack / ready / alive / config / config-ack / close`)
- **Data plane (LAN)**: dedicated binary WebSocket on the same port, path `/camera-stream/{sessionId}`
- **Data plane (relay)**: degraded mode — `camera.v1.frame` as JSON on the main WebSocket
- **One session = one camera.** To stream multiple cameras simultaneously, open multiple sessions.
- **Codec negotiated at open time**: the controller offers a preference list; the host picks one. Frames are self-describing — the binary header's `codec` field identifies the encoding.
- **Alive-controlled streaming**: the host streams only while the controller sends periodic `camera.v1.alive` heartbeats on the control plane.

## Security

- **sessionId** (UUIDv4): public session identifier, used to correlate messages across control and data planes.
- **streamToken** (opaque string, generated by host): one-time credential for the data connection. Included in `camera.v1.open-ack` and consumed on first use.

When the controller connects to the LAN data WebSocket (`/camera-stream/{sessionId}?token={streamToken}`), the host validates the token and immediately invalidates it. Subsequent connection attempts with the same token are rejected. The path is deregistered when the session ends.

## Binary Data Frame Format (LAN data connection)

All frames on the dedicated data WebSocket use binary mode with the following header:

```
Offset  Size  Field
0       1B    version (0x01)
1       1B    kind
2       1B    codec
3       1B    flags
4       4B    sequence number (big-endian u32)
8       4B    timestamp_ms — milliseconds since session start (big-endian u32)
12      4B    payload length (big-endian u32)
16      ...   payload (kind-dependent)
```

Total header: 16 bytes.

### Kind Values

| Value | Name   | Direction        | Payload                                     |
|-------|--------|------------------|---------------------------------------------|
| 0x01  | frame  | host → controller | Encoded video frame bytes                  |
| 0x02  | cancel | either direction  | Optional UTF-8 reason string (see [Reason Format](../README.md#reason-format)) |

### Codec Values

| Value | Name    | Description                                  |
|-------|---------|----------------------------------------------|
| 0x01  | `jpeg`  | JPEG still image                             |
| 0x02  | `h264`  | H.264/AVC Annex B access unit                |
| 0x03  | `webp`  | WebP image                                   |

The codec field in frame headers MUST match the `negotiatedCodec` from `camera.v1.open-ack`. New codec values require a Business Protocol minor version bump.

### H.264 Payload Format

When the negotiated codec is `h264`:

- Each frame payload MUST contain exactly one complete H.264 access unit in Annex B byte-stream format.
- Every NAL unit MUST be prefixed with an Annex B start code.
- A frame marked with the `keyframe` flag MUST contain an IDR picture and all SPS/PPS parameter sets required to decode it. The parameter sets MUST precede the picture's VCL NAL units in the same access unit.
- The first frame of a stream MUST be a keyframe. After a configuration change that alters the encoded video format, the first frame in the new format MUST also be a keyframe.

Platform codec output that uses a different representation, such as length-prefixed AVC samples or separately supplied parameter sets, MUST be normalized to this format before transmission.

### Flags

| Bit | Name     | Description                                              |
|-----|----------|----------------------------------------------------------|
| 0   | keyframe | Set when this frame can be decoded independently (I-frame or full JPEG). Receivers that have lost sync MUST wait for the next frame with this bit set before rendering. |
| 1–7 | reserved | MUST be 0 on send; MUST be ignored on receive            |

Data connection liveness is handled by WebSocket-level ping/pong; no dedicated alive frame kind is defined.

## Message Types

| Type | Direction | Description |
|------|-----------|-------------|
| `camera.v1.list` | controller → host | Request available cameras |
| `camera.v1.list-result` | host → controller | Return camera list |
| `camera.v1.open` | controller → host | Open a streaming session |
| `camera.v1.open-ack` | host → controller | Accept or reject the session |
| `camera.v1.ready` | controller → host | Confirm the selected data transport |
| `camera.v1.alive` | controller → host | Streaming liveness heartbeat |
| `camera.v1.config` | controller → host | Adjust stream parameters mid-session |
| `camera.v1.config-ack` | host → controller | Confirm or reject parameter change |
| `camera.v1.close` | bidirectional | Close a session |
| `camera.v1.frame` | host → controller | Video frame — relay mode only |

## camera.v1.list

Request the list of cameras available on the host device.

```json
{
  "type": "camera.v1.list",
  "payload": {}
}
```

No payload fields required.

### camera.v1.list-result

```json
{
  "type": "camera.v1.list-result",
  "payload": {
    "cameras": [
      {
        "cameraId": "com.apple.avfoundation.avcapturedevice.built-in_video:0",
        "label": "FaceTime HD Camera",
        "position": "front",
        "capabilities": {
          "resolutions": [
            { "width": 1280, "height": 720 },
            { "width": 640, "height": 480 }
          ],
          "fpsRange": { "min": 1, "max": 30 }
        }
      }
    ]
  }
}
```

| Field   | Type        | Description                   |
|---------|-------------|-------------------------------|
| cameras | CameraEntry[] | Available cameras on the host |

**CameraEntry**

| Field        | Type   | Description                                                                 |
|--------------|--------|-----------------------------------------------------------------------------|
| cameraId     | string | Opaque device identifier, stable for the lifetime of the session            |
| label        | string | Human-readable camera name                                                  |
| position     | string/null | `"front"`, `"back"`, `"external"`, or `null` when not applicable       |
| capabilities | Capabilities/null | Supported parameters. `null` if the host cannot determine them       |

**Capabilities**

| Field       | Type             | Description                                                       |
|-------------|------------------|-------------------------------------------------------------------|
| resolutions | Resolution[]     | Supported resolutions, ordered from highest to lowest             |
| fpsRange    | FpsRange         | Supported frame rate range                                        |

**Resolution**

| Field  | Type    | Description          |
|--------|---------|----------------------|
| width  | integer | Frame width in pixels  |
| height | integer | Frame height in pixels |

**FpsRange**

| Field | Type    | Description               |
|-------|---------|---------------------------|
| min   | integer | Minimum frames per second |
| max   | integer | Maximum frames per second |

### Notes

- The host SHOULD return all cameras it can enumerate, including those currently in use by other applications.
- If a camera becomes unavailable between `camera.v1.list` and `camera.v1.open`, the host MUST reject the open with `colink:camera.not_available.v1`.

## camera.v1.open

Request a new streaming session on the host.

```json
{
  "type": "camera.v1.open",
  "payload": {
    "sessionId": "a3f1e2d4-58cc-4372-b567-0e02b2c3d479",
    "cameraId": "com.apple.avfoundation.avcapturedevice.built-in_video:0",
    "preferredCodecs": ["h264", "webp", "jpeg"],
    "preferredWidth": 1280,
    "preferredHeight": 720,
    "preferredFps": 30
  }
}
```

| Field           | Type     | Required | Description                                                           |
|-----------------|----------|----------|-----------------------------------------------------------------------|
| sessionId       | string   | Yes      | UUIDv4 generated by the controller; unique per connection             |
| cameraId        | string   | Yes      | Camera identifier from `camera.v1.list-result`                        |
| preferredCodecs | string[] | Yes      | Ordered list of codec names the controller supports; host picks the first it also supports |
| preferredWidth  | integer  | No       | Desired frame width in pixels; host MAY use a different resolution    |
| preferredHeight | integer  | No       | Desired frame height in pixels; host MAY use a different resolution   |
| preferredFps    | integer  | No       | Desired frame rate; host MAY use a different rate                     |

### Notes

- The controller MUST NOT reuse a `sessionId` within the same connection.
- If `preferredCodecs` contains no codec the host supports, the host MUST reject with `colink:camera.no_common_codec.v1`.

## camera.v1.open-ack

Sent by the host in response to `camera.v1.open`. The `correlationId` in the enclosing transport message MUST reference the `id` of the message that carried the originating `camera.v1.open`.

**Accepted:**

```json
{
  "type": "camera.v1.open-ack",
  "payload": {
    "sessionId": "a3f1e2d4-58cc-4372-b567-0e02b2c3d479",
    "accepted": true,
    "negotiatedCodec": "h264",
    "width": 1280,
    "height": 720,
    "fps": 30,
    "streamToken": "one-time-stream-token"
  }
}
```

**Rejected:**

```json
{
  "type": "camera.v1.open-ack",
  "payload": {
    "sessionId": "a3f1e2d4-58cc-4372-b567-0e02b2c3d479",
    "accepted": false,
    "reason": "colink:camera.rejected.v1",
    "message": "Camera access denied on this device"
  }
}
```

| Field           | Type    | Required       | Description                                                              |
|-----------------|---------|----------------|--------------------------------------------------------------------------|
| sessionId       | string  | Yes            | Echoed from the request                                                  |
| accepted        | boolean | Yes            | Whether the session was opened                                           |
| negotiatedCodec | string  | When accepted  | The codec the host will use for this session                             |
| width           | integer | When accepted  | Actual frame width in pixels                                             |
| height          | integer | When accepted  | Actual frame height in pixels                                            |
| fps             | integer | When accepted  | Actual frame rate                                                        |
| streamToken     | string  | When LAN is offered | One-time credential for the LAN data WebSocket connection; omitted when the host requires relay mode |
| reason          | string  | When rejected  | Structured reason code                                                   |
| message         | string  | When rejected  | Human-readable description for logging/debugging                         |

### Well-Known Rejection Reasons

| Reason | Description |
|--------|-------------|
| `colink:camera.rejected.v1` | Host declined to open the session (e.g. feature disabled, permission denied) |
| `colink:camera.not_available.v1` | The specified `cameraId` does not exist or is unavailable |
| `colink:camera.no_common_codec.v1` | No codec in `preferredCodecs` is supported by the host |
| `colink:camera.session_conflict.v1` | `sessionId` duplicates an existing active session |
| `colink:camera.generic.v1` | Generic failure not covered by a specific reason |

### Notes

- The host decides whether to offer a LAN data connection. When offering LAN, it MUST register the data WebSocket path before sending an accepted `camera.v1.open-ack` containing `streamToken`.
- When the host requires relay mode, it MUST omit `streamToken` and MUST NOT register a data WebSocket path for the session.
- The host MUST NOT begin streaming until it has received both `camera.v1.ready` and the first `camera.v1.alive` for the session.

## camera.v1.ready

Sent by the controller after `camera.v1.open-ack` to confirm the transport that will carry video frames.

```json
{
  "type": "camera.v1.ready",
  "payload": {
    "sessionId": "a3f1e2d4-58cc-4372-b567-0e02b2c3d479",
    "transport": "lan"
  }
}
```

| Field     | Type   | Description                                      |
|-----------|--------|--------------------------------------------------|
| sessionId | string | The session whose transport has been selected   |
| transport | string | `"lan"` or `"relay"`                           |

### Notes

- If `camera.v1.open-ack` omits `streamToken`, the controller MUST select `relay`.
- If `camera.v1.open-ack` includes `streamToken`, the controller SHOULD first attempt the LAN data WebSocket. After the connection succeeds, it MUST send `camera.v1.ready` with `transport: "lan"`. If the connection attempt fails, it MUST send `camera.v1.ready` with `transport: "relay"`.
- The host MUST accept `transport: "lan"` only after the session's data WebSocket has connected and its `streamToken` has been validated.
- When the controller selects `relay`, the host MUST invalidate any unused `streamToken` and deregister the session's data WebSocket path.
- The selected transport cannot change during the session. The host MUST silently ignore subsequent `camera.v1.ready` messages for the same session and `camera.v1.ready` messages for an unknown `sessionId`.

---

## camera.v1.alive

Sent periodically by the controller to indicate it is still receiving the stream. The host streams only to controllers whose `alive` heartbeat remains within the timeout window.

```json
{
  "type": "camera.v1.alive",
  "payload": {
    "sessionId": "a3f1e2d4-58cc-4372-b567-0e02b2c3d479"
  }
}
```

| Field     | Type   | Description                        |
|-----------|--------|------------------------------------|
| sessionId | string | The session this heartbeat is for  |

### Notes

- Controllers SHOULD send `camera.v1.alive` at a fixed interval (recommended: 5s).
- The host SHOULD stop streaming to a controller if no `camera.v1.alive` is received for that session within a timeout (recommended: 15s), and SHOULD release the camera resource if no other sessions are active.
- The host SHOULD resume streaming when a new `camera.v1.alive` arrives for a paused session.
- The host MUST silently ignore `camera.v1.alive` for an unknown `sessionId`.

## camera.v1.config

Sent by the controller to adjust stream parameters mid-session.

```json
{
  "type": "camera.v1.config",
  "payload": {
    "sessionId": "a3f1e2d4-58cc-4372-b567-0e02b2c3d479",
    "width": 640,
    "height": 480,
    "fps": 15
  }
}
```

| Field     | Type    | Required | Description                                      |
|-----------|---------|----------|--------------------------------------------------|
| sessionId | string  | Yes      | The session to reconfigure                       |
| width     | integer | No       | New desired frame width in pixels                |
| height    | integer | No       | New desired frame height in pixels               |
| fps       | integer | No       | New desired frame rate                           |

### Notes

- At least one of `width`, `height`, or `fps` MUST be present.
- The host MUST respond with `camera.v1.config-ack`. The `correlationId` in the enclosing transport message MUST reference the `id` of the message that carried the originating `camera.v1.config`.
- The host MUST silently ignore `camera.v1.config` for an unknown `sessionId`.

---

## camera.v1.config-ack

Sent by the host in response to `camera.v1.config`.

**Applied:**

```json
{
  "type": "camera.v1.config-ack",
  "payload": {
    "sessionId": "a3f1e2d4-58cc-4372-b567-0e02b2c3d479",
    "applied": true,
    "width": 640,
    "height": 480,
    "fps": 15
  }
}
```

**Rejected:**

```json
{
  "type": "camera.v1.config-ack",
  "payload": {
    "sessionId": "a3f1e2d4-58cc-4372-b567-0e02b2c3d479",
    "applied": false,
    "reason": "colink:camera.config_rejected.v1",
    "message": "Requested resolution is not supported"
  }
}
```

| Field     | Type    | Required      | Description                                                                 |
|-----------|---------|---------------|-----------------------------------------------------------------------------|
| sessionId | string  | Yes           | Echoed from the request                                                     |
| applied   | boolean | Yes           | Whether the new parameters were applied                                     |
| width     | integer | When applied  | Actual frame width after reconfiguration (may differ from requested)        |
| height    | integer | When applied  | Actual frame height after reconfiguration                                   |
| fps       | integer | When applied  | Actual frame rate after reconfiguration                                     |
| reason    | string  | When rejected | Structured reason code                                                      |
| message   | string  | When rejected | Human-readable description for logging/debugging                            |

### Well-Known Rejection Reasons

| Reason | Description |
|--------|-------------|
| `colink:camera.config_rejected.v1` | Host cannot apply the requested parameters |
| `colink:camera.generic.v1` | Generic failure not covered by a specific reason |

### Notes

- When `applied: true`, the host applies the new parameters starting from the next keyframe. Frames already in flight with the old parameters are sent as-is. For H.264, that keyframe MUST satisfy the parameter-set requirements in [H.264 Payload Format](#h264-payload-format).
- The actual values in the ack reflect what was applied, which MAY differ from what was requested if the host clamped or adjusted the values.

## camera.v1.close

Closes a streaming session. Either side MAY send this message.

**Host-initiated (camera error or resource reclaimed):**

```json
{
  "type": "camera.v1.close",
  "payload": {
    "sessionId": "a3f1e2d4-58cc-4372-b567-0e02b2c3d479",
    "reason": "colink:camera.device_lost.v1",
    "message": "Camera disconnected"
  }
}
```

**Controller-initiated (user closed the viewer):**

```json
{
  "type": "camera.v1.close",
  "payload": {
    "sessionId": "a3f1e2d4-58cc-4372-b567-0e02b2c3d479",
    "reason": null
  }
}
```

| Field     | Type        | Required | Description                                                        |
|-----------|-------------|----------|--------------------------------------------------------------------|
| sessionId | string      | Yes      | The session to close                                               |
| reason    | string/null | No       | Structured reason code when host-initiated; `null` or omitted for controller-initiated close |
| message   | string      | No       | Human-readable description for logging/debugging                   |

### Well-Known Host-Initiated Reasons

| Reason | Description |
|--------|-------------|
| `colink:camera.device_lost.v1` | Camera hardware disconnected or became unavailable |
| `colink:camera.alive_timeout.v1` | No `camera.v1.alive` received within the timeout window |
| `colink:camera.generic.v1` | Generic failure not covered by a specific reason |

### Notes

- After either side sends `camera.v1.close`, no further `camera.v1.ready`, `camera.v1.frame`, `camera.v1.config`, `camera.v1.config-ack`, or `camera.v1.alive` messages MUST be sent for that `sessionId`.
- The host MUST close the data WebSocket connection and release the camera resource upon sending or receiving `camera.v1.close`.
- The host MUST silently ignore `camera.v1.close` for an unknown `sessionId`.

## camera.v1.frame (relay mode only)

Carries a single encoded video frame on the main WebSocket in relay mode.

```json
{
  "type": "camera.v1.frame",
  "payload": {
    "sessionId": "a3f1e2d4-58cc-4372-b567-0e02b2c3d479",
    "codec": "jpeg",
    "keyframe": true,
    "sequence": 42,
    "timestampMs": 1400,
    "data": "base64encodedframedata..."
  }
}
```

| Field       | Type    | Description                                                              |
|-------------|---------|--------------------------------------------------------------------------|
| sessionId   | string  | The session this frame belongs to                                        |
| codec       | string  | Codec name matching the session's `negotiatedCodec`                      |
| keyframe    | boolean | Whether this frame can be decoded independently                          |
| sequence    | integer | Monotonically increasing sequence number (0-based)                       |
| timestampMs | integer | Milliseconds since session start                                         |
| data        | string  | Base64-encoded frame bytes                                               |

### Notes

- `camera.v1.frame` MUST NOT be sent over a LAN data WebSocket; it is only used in relay mode.
- When `codec` is `h264`, the decoded bytes of `data` MUST follow [H.264 Payload Format](#h264-payload-format).
- Implementations SHOULD reduce `preferredFps` and resolution in relay mode to limit bandwidth on the relay channel.
- The controller MUST silently ignore frames for an unknown `sessionId`.

## Connection Disconnect Behavior

When the underlying transport closes for any reason:

- The host MUST close all active data WebSocket connections and release all camera resources held by sessions originating from the disconnected controller.
- The controller MUST treat all open sessions as closed upon disconnect.
- No `camera.v1.close` messages are exchanged for sessions implicitly closed by a disconnect.

**Cloud-relay topology:** When the host receives a `device.offline` event for the controller, it MUST treat this as equivalent to a connection disconnect and terminate all active camera sessions created by that controller.

**LAN data connection:** If the data WebSocket disconnects while the control WebSocket remains open, the host MUST treat the session as closed and send `camera.v1.close` on the control plane.

## Flow

### LAN (dedicated data connection)

```
Controller                   Main WS                    Host
  |                                                       |
  |--- camera.v1.list -------------------------------------> |
  |<-- camera.v1.list-result ----------------------------- |
  |                                                       |
  |--- camera.v1.open -------------------------------------> |  { sessionId, cameraId, preferredCodecs, ... }
  |<-- camera.v1.open-ack -------------------------------- |  { sessionId, accepted: true, negotiatedCodec, streamToken, ... }
  |                                                       |  (/camera-stream/{sessionId} already registered)
  |===== connect ws://{ip}:{port}/camera-stream/{id}?token={t} ===>|  (dedicated binary WS)
  |                                                       |
  |--- camera.v1.ready -----------------------------------> |  { sessionId, transport: "lan" }
  |--- camera.v1.alive -----------------------------------> |  { sessionId }  (every 5s)
  |                                                       |
  |<== [frame, seq=0, keyframe=1] == data WS ============ |
  |<== [frame, seq=1] =========== data WS ============== |
  |<== [frame, seq=2] =========== data WS ============== |
  |                                                       |
  |--- camera.v1.config -----------------------------------> |  { sessionId, width: 640, height: 480 }
  |<-- camera.v1.config-ack ------------------------------ |  { sessionId, applied: true, width: 640, height: 480, fps: 30 }
  |                                                       |
  |<== [frame, seq=3, keyframe=1] == data WS ============ |  (first frame at new resolution)
  |                                                       |
  |--- camera.v1.close ------------------------------------> |  { sessionId }
  |===== data WS closed ================================= |
```

### Cloud Relay (degraded mode)

```
Controller                   Main WS                    Host
  |                                                       |
  |--- camera.v1.list -------------------------------------> |
  |<-- camera.v1.list-result ----------------------------- |
  |                                                       |
  |--- camera.v1.open -------------------------------------> |  { sessionId, cameraId, preferredCodecs, ... }
  |<-- camera.v1.open-ack -------------------------------- |  { sessionId, accepted: true, negotiatedCodec, ... }  (no streamToken)
  |                                                       |
  |--- camera.v1.ready -----------------------------------> |  { sessionId, transport: "relay" }
  |--- camera.v1.alive -----------------------------------> |  { sessionId }  (every 5s)
  |<-- camera.v1.frame ----------------------------------- |  { sessionId, codec, keyframe: true, sequence: 0, data }
  |<-- camera.v1.frame ----------------------------------- |  { sessionId, codec, keyframe: false, sequence: 1, data }
  |                                                       |
  |--- camera.v1.close ------------------------------------> |  { sessionId }
```

### LAN Attempt Fallback to Relay

```
Controller                   Main WS                    Host
  |                                                       |
  |--- camera.v1.open -------------------------------------> |
  |<-- camera.v1.open-ack -------------------------------- |  { accepted: true, streamToken, ... }
  |                                                       |
  |===== LAN data WebSocket connection failed ===========X |
  |--- camera.v1.ready -----------------------------------> |  { sessionId, transport: "relay" }
  |                                                       |  (invalidates token and deregisters data path)
  |--- camera.v1.alive -----------------------------------> |
  |<-- camera.v1.frame ----------------------------------- |
  |                                                       |
  |--- camera.v1.close ------------------------------------> |
```

## Version Compatibility

- `camera.v1.*` requires Business Protocol Version 1.10.0 or later.
- Before sending `camera.v1.list` or `camera.v1.open`, a controller MUST verify that the target's advertised `businessVersion` is valid, has the same major version, and is at least 1.10.0. If the version is missing, malformed, incompatible, or too old, the controller MUST NOT send these messages.
- A host that predates Version 1.10.0 treats incoming `camera.v1.*` messages as unknown Business messages and silently ignores them according to the standard forward-compatibility rule.
