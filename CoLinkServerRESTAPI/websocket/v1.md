# Cloud WebSocket

Real-time communication channel between device and cloud server.

## Connection

```
wss://server/ws/v1?ticket=<ticket>&businessVersion=<semver>
```

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| ticket | string | yes | One-time ticket from `POST /api/v1/ws/ticket` |
| businessVersion | string | yes | Business protocol version (semver) to advertise to peers |

Obtain a `ticket` via `POST /api/v1/ws/ticket` (see `ticket.md`). The ticket is single-use and expires after 30 seconds.

## Message Envelope

All messages use a unified JSON format:

```json
{
  "id": "a1b2c3d4-e5f6-7890-abcd-ef0123456789",
  "type": "message.type",
  "from": "deviceId",
  "to": "deviceId | null",
  "payload": {},
  "timestamp": 1716451200000
}
```

| Field     | Type        | Description                                 |
|-----------|-------------|---------------------------------------------|
| id        | string      | Message UUID (for dedup, ACK, correlation)  |
| type      | string      | Message type identifier                     |
| from      | string/null | Sender device ID (server fills for inbound) |
| to        | string/null | Target device ID (for relay only, null for system events) |
| payload   | object      | Message-specific data                       |
| timestamp | number      | Unix timestamp in milliseconds              |

## Lifecycle

1. Client obtains ticket via `POST /api/v1/ws/ticket`
2. Client connects with `ticket` and `businessVersion` in query string
3. Server validates ticket → connection established → device marked online
4. Server broadcasts `device.online` to same-account devices
5. Server sends `device.online` for every other currently online device to the newly connected client (catch-up). These messages are identical in format to real-time broadcasts; the client handles them uniformly.
6. Client sends `ping` every 30s to keep alive
7. On disconnect → server broadcasts `device.offline`

## Server → Device Messages

### device.online

```json
{
  "id": "b2c3d4e5-f6a7-8901-bcde-f01234567890",
  "type": "device.online",
  "from": "660e8400-...",
  "to": null,
  "payload": {
    "name": "My Phone",
    "type": "android",
    "businessVersion": "1.0.0"
  },
  "timestamp": 1716451200000
}
```

| Field | Type | Description |
|-------|------|-------------|
| name | string | Device display name |
| type | string | Device platform type |
| businessVersion | string | Business protocol version (semver) reported via WebSocket query parameter |

The server only distributes `businessVersion` as device presence metadata. It does not interpret business protocol messages and does not enforce business-version compatibility for `relay` or `broadcast`. Before sending business messages, the client is responsible for checking the target device's advertised `businessVersion` and choosing a compatible payload, degrading behavior, or skipping the target.

### device.offline

```json
{
  "id": "c3d4e5f6-a7b8-9012-cdef-012345678901",
  "type": "device.offline",
  "from": "660e8400-...",
  "to": null,
  "payload": {},
  "timestamp": 1716451200000
}
```

### broadcast

Message broadcasted from another device:

```json
{
  "id": "b8c9d0e1-f2a3-4567-bcde-567890123456",
  "type": "broadcast",
  "from": "660e8400-...",
  "to": null,
  "payload": {
    "type": "clipboard.v1.sync",
    "payload": { "contentType": "text/plain", "content": "Hello from device A" }
  },
  "timestamp": 1716451200000
}
```

Inner `payload` is a business protocol message, same as `relay`.

### relay (inbound)

Message relayed from another device:

```json
{
  "id": "d4e5f6a7-b8c9-0123-defa-123456789012",
  "type": "relay",
  "from": "660e8400-...",
  "to": "770e8400-...",
  "payload": {
    "type": "clipboard.v1.sync",
    "payload": { "contentType": "text/plain", "content": "Hello" }
  },
  "timestamp": 1716451200000
}
```

Inner `payload` is a business protocol message.

### pong

```json
{ "id": "e5f6a7b8-c9d0-1234-efab-234567890123", "type": "pong", "timestamp": 1716451200000 }
```

## Device → Server Messages

### ping

```json
{ "id": "f6a7b8c9-d0e1-2345-fabc-345678901234", "type": "ping" }
```

### relay (outbound)

Request server to forward a message to another device:

```json
{
  "id": "a7b8c9d0-e1f2-3456-abcd-456789012345",
  "type": "relay",
  "to": "770e8400-...",
  "payload": {
    "type": "message.v1.text",
    "payload": { "messageId": "88a3c4d5-e6f7-8901-abcd-ef0123456789", "text": "hi" }
  }
}
```

Server fills `from` and delivers. If target is offline, message is dropped (no offline queue in MVP).

### broadcast

Request server to broadcast a message to all other devices of the same account:

```json
{
  "id": "b8c9d0e1-f2a3-4567-bcde-567890123456",
  "type": "broadcast",
  "payload": {
    "type": "clipboard.v1.sync",
    "payload": { "contentType": "text/plain", "content": "Broadcast clipboard" }
  }
}
```

**Behavior**:
- Server broadcasts to all online devices of the same user **except** the sender
- Offline devices do not receive the message
